The AI worker, in three phases.

How Ananta's six AI teachers actually teach, answer doubts, generate notes, and find their own voice — without us building our own GPT. Pragmatic in Phase 1, defensible by Phase 3.

🇮🇳 Indic-first 💸 ₹0 LLM cost at MVP 🔓 Open-source spine 🎙 Custom TTS per teacher 📈 RAG → fine-tune → custom 70B

🧭 The position we're taking

  1. Don't train a foundation model. A model the size of GPT-4 takes $10M and a research team to train. That's a different business. Use what already exists.
  2. The knowledge is ours, the model is rented. Off-the-shelf LLMs become "Ananta teachers" through retrieval-augmented generation (RAG) over our own lesson scripts, KSEEB textbooks, and accumulated doubt-answer pairs. The model is interchangeable; the knowledge base is the moat.
  3. Persona via prompt engineering, not training. Each teacher has a system prompt + few-shot examples + locked vocabulary. Consistent voice without a single training run.
  4. One thing IS worth training early: TTS voices. Six fine-tuned voices on AI4Bharat's IndicTTS — cheap (~₹50k each), permanent brand asset, can't be replicated by an OpenAI wrapper.
  5. Phase in the upgrades. RAG (now) → fine-tune small open model (at 500 users) → custom 70B (at 5,000 users). Each phase pays for the next.
  6. Cost ceiling: zero rupees of LLM API spend until paying users exist. Groq's dev tier covers it. If we cross the ceiling, we have revenue to offset.

01 What the AI worker does

The AI worker is a separate FastAPI + Celery service that handles everything LLM-shaped. Splitting it from the main API means the slow stuff (lesson rendering, TTS generation, doubt answering) never blocks transactional requests like signups and payments. The student's browser talks to the NestJS API; the API delegates AI work to this service.

Responsibilities
  • Lesson generation — produce the full script, the visual scene plan, and the Manim/Remotion animation code for one chapter, in both English and Kanglish.
  • Lesson narration (TTS) — turn that script into an audio file using the teacher's fine-tuned voice.
  • Live doubt answering — during a class, answer typed and voiced student questions in real time, in the teaching teacher's persona.
  • Notes generation — produce the auto-generated lesson notes that students download after class.
  • Quiz generation — write the 6-10 question quiz that appears at the end of each class, plus answer keys and explanations.
  • Memory Deck card generation — extract spaced-repetition flashcards (definition, formula, key-fact) from each completed lesson.
  • Sunday weekly test paper generation — assemble the 50-question paper from the week's chapters.
  • Catch-up plan generation — for late-joining or behind-the-cohort students, produce a personalised daily watch order.
  • Parent-report writing — compose the weekly WhatsApp digest in the parent's preferred language.

Models used in Phase 1
Phase 1 Free at MVP
  • Groq Llama 3.3 70B Versatile — the workhorse. Used for English-medium lesson script writing, quiz generation, parent reports, and most doubt answering. ~500 tokens/sec inference speed. Free dev tier: 30 req/min, 14,000 req/day.
  • Sarvam-1 (2B, AI4Bharat) — Kanglish doubt answering and Kannada-medium lesson narration script polish. The only model that handles Kannada-English code-mixing well.
  • OpenRouter free tier (DeepSeek, Gemini 2.0 Flash, Llama 3.1 70B) — fallback when Groq rate-limits. Set as automatic failover in the worker code.
  • BGE-M3 embeddings — multilingual embedding model (1024-d vectors). Free to self-host. Used for both English and Kannada/Hindi knowledge base. Matches the vector(1024) column in the Prisma schema.

03 The RAG pipeline, step by step

When a student asks a doubt mid-class, here's everything that happens in the ~800 ms before the answer starts streaming back.

Doubt → Answer, in eight steps
  1. The doubt arrives. Student types: "Why does magnesium burn so brightly?" Browser POSTs to /doubts. NestJS API validates, persists the doubt to the Postgres Doubt table with status PENDING, then forwards to AI worker.
  2. Embed the query. AI worker runs the doubt through BGE-M3, producing a 1024-dimensional vector. ~30 ms on the Oracle ARM CPU.
  3. Retrieve top-K relevant chunks. Worker runs SELECT ... ORDER BY embedding <=> $1 LIMIT 8 against the lesson-chunk table, filtered by current subject and the chapters covered so far. ~40 ms with pgvector's HNSW index.
  4. Score and filter retrieved chunks. Worker re-ranks the 8 chunks with a small cross-encoder (BGE-Reranker). Drops anything below threshold. Usually 3-5 chunks survive.
  5. Assemble the prompt. Template: {teacher_system_prompt} + {few_shot_examples} + {retrieved_chunks} + {the_student_question} + {output_format_instruction}.
  6. Call the LLM. Streaming Groq completion (Llama 3.3 70B). Worker proxies the stream back to the API, which streams it to the browser. Time-to-first-token: ~300 ms.
  7. Persist the answer. While streaming, also accumulate the full answer. When the stream ends, write it to the Doubt row with status ANSWERED, generate an embedding for the answer, store both in pgvector. This future doubt may now retrieve THIS answer.
  8. Log quality signal. A "Was this helpful?" thumbs widget appears under the answer. Whatever the student clicks is logged with the doubt — feeds the Phase 2 fine-tuning dataset.

The system prompt skeleton

You are {teacher_name}, the AI teacher for {subject} on Ananta.
You teach {medium}-medium Karnataka State Board, Grade {grade}.

TEACHING STYLE:
{persona_block}

VOICE / VOCABULARY:
- Use these phrases naturally: {locked_phrases}
- Never use: {forbidden_phrases}
- {tone_rules}

CONTEXT YOU HAVE ACCESS TO (from Ananta's knowledge base):
{retrieved_chunks}

RULES:
- Answer ONLY using the context above + standard KSEEB syllabus knowledge.
- If the context doesn't contain the answer, say "Let me note this — I'll
  cover it in detail in a future class" and add it to the doubt-followup queue.
- Cite the lesson the fact came from: e.g., "(from Ch.2 — Reactions)"
- Match the student's medium: if they wrote in Kanglish, answer in Kanglish.
- Keep answers under 200 words unless they explicitly asked for detail.

STUDENT'S QUESTION:
{student_question}

YOUR ANSWER (as {teacher_name}):

04 Persona system — how Vihaan stays Vihaan

Six teachers, six distinct voices. Each lives as a YAML file in apps/ai-worker/personas/. Loading the file produces the system prompt and the few-shot examples for that teacher. 

# apps/ai-worker/personas/vihaan.yaml
name: Vihaan
subject: Science
day: Tuesday
description: Playful demonstrator. Loves chemistry. Treats every lesson as a small experiment.

teaching_style: |
  You teach Science with energy. You begin with a demonstration —
  "let's watch this happen first, then I'll explain why."
  You use everyday Karnataka examples: rust on a cycle, baking soda
  in idli batter, sweat evaporating after running. You're careful
  to distinguish observation from inference.

vocabulary:
  use_naturally:
    - "Let's see what happens..."
    - "Watch closely."
    - "What did you notice?"
    - "This is the same thing that happens when..."
    - "ಗಮನಿಸಿ" (when in Kanglish)
  avoid:
    - "obviously"
    - "as you all know"
    - "trivially"

tone:
  formality: 2  # 1 (casual) to 5 (formal)
  energy: 4
  patience: 5

example_answers:
  - question: "Why does iron rust?"
    answer: "Look at any old gate left out in the rain — you'll see brown patches.
    That's rust. It happens when iron meets oxygen and water at the same time.
    The iron slowly turns into iron oxide (Fe₂O₃). This is an oxidation reaction
    — the iron is losing electrons to the oxygen. (from Ch.1 — Chemical Reactions)"
  - question: "ಸರ್, electrolysis ennu mean?"  # Kanglish
    answer: "Electrolysis andre — water-ge current pass madidaga, water hydrogen
    mattu oxygen-aagi separate aaguttade. 2H₂O → 2H₂ + O₂. Battery-ind electricity
    bandu, water-na break maaduttade. Try maadi mane-li battery + salt water + 2 nails
    use maadi — bubbles bartve. (from Ch.3 — Electricity & Chemicals)"
The trick: Personas don't need to be fine-tuned in. A well-written 60-line YAML + 5 example answers is enough to make a 70B model reliably write as Vihaan instead of Adhvara. We've used this same pattern in production at other companies and it holds up at scale.

05 Knowledge base & embeddings

The RAG knowledge base is everything the model is allowed to know. We control what goes in.

What gets embedded into pgvector
  • Every lesson script — chunked by paragraph. Embedded immediately after lesson generation. Fields stored: subject, chapter, lesson_id, medium, chunk_text, embedding.
  • The full KSEEB Grade 10/11/12 textbooks — one-time embedding job. ~6,000 chunks across all subjects. Stored with source: "kseeb_textbook".
  • Every doubt + answer pair — once an answer is approved (student gave thumbs-up OR no thumbs-down within 24h), both question and answer get embedded. Doubt answers become retrieval targets — meaning, when a new student asks a similar question, we find the existing high-quality answer first.
  • Memory Deck card front + back — embedded so semantically-similar cards are deduplicated.
  • Marketing / FAQ corpus — small (~200 chunks), used only by the support chatbot.
What does NOT get embedded
  • General web content — too noisy, no way to verify quality.
  • Other tutoring platforms' content — legal + brand risk.
  • Random YouTube transcripts — quality varies wildly.
  • Student personal data — privacy. Doubts get embedded ANONYMIZED (student-id stripped before embed).

The chunking rules (this matters more than people think)

  • Target chunk size: ~400 tokens (≈ 1500 characters).
  • Always include the chapter title + section title as a header on every chunk. ("Ch.2 § Types of Reactions — ...").
  • Overlap chunks by ~50 tokens so a paragraph crossing chunk boundary still retrieves cleanly.
  • Code/formula chunks (Manim scripts, balanced equations) get their own embedding type.
  • Multilingual chunks: store one English-translated copy AND the original, both embedded. Helps a Kanglish doubt retrieve from an English-language source.

06 TTS — six voices, fine-tuned, owned

The one place we DO train in Phase 1. Each AI teacher gets their own fine-tuned TTS voice, so "Vihaan" sounds like Vihaan whether he's teaching, answering, or chatting — and no competitor wrapping the same off-the-shelf API can copy it.

The TTS production process
  1. Record reference audio. Hire one voice artist per teacher for a 4-hour session (priced ~₹3-8k/hr in Bengaluru). They record ~500 lines covering the full phonetic range plus typical teaching cadence in the teacher's medium.
  2. Clean + align. Standard audio prep — strip noise, normalize loudness, align transcripts.
  3. Fine-tune AI4Bharat IndicTTS adapter. Train a LoRA adapter on the recorded data. ~2 hours on a rented T4 GPU. ₹500-1000 in compute per teacher.
  4. Evaluate. Generate 50 held-out test sentences. Human listener (priyanka) rates each. If anything sounds off, iterate.
  5. Deploy. Adapter weights stored in R2. AI worker loads the right adapter at inference time based on which teacher is speaking.
Total cost per teacher (one-time)
Voice artist: ₹15k-25k · GPU training: ₹1k · Engineering time: 2 days. ~₹25k-40k per teacher × 6 = ₹1.5-2.4 lakh for all six. Pays for itself in brand value the moment a student says "the Vihaan voice is what makes the lessons feel real."
Fallback: Generic AI4Bharat IndicTTS (no fine-tuning) is the fallback if a teacher's adapter fails to load. Students never hear a missing voice — they hear a slightly more generic version of it.

  • Script generation — Groq Llama 3.3 70B with the teacher's persona prompt + chapter content from KSEEB textbook (retrieved from knowledge base) + the previous N lessons' summaries (so it knows where students are). ~30 sec per lesson.
  • Animation plan — second LLM call asks "list the visual scenes for this lesson". Output is a JSON schedule of Manim or Remotion scenes with timing.
  • Animation rendering — Manim CLI or Remotion CLI invoked by Celery. Renders MP4 at 1080p. ~5 minutes per lesson on the Oracle ARM (Manim is CPU-bound and ARM-friendly).
  • TTS rendering — Stream the script through the teacher's fine-tuned IndicTTS adapter sentence-by-sentence. Concatenate WAVs. ~3 minutes per lesson.
  • Final mux — FFmpeg combines the animation video with the TTS audio. Adds captions (generated from the script).

08 Live doubt answering — the hard one

During the 7-8 PM class, the AI teacher is answering doubts in near-real-time. This needs to feel instant (under 1 second to first token) and consistent in voice. Here are the moving parts.

  • WebSocket from browser — keeps a persistent connection so the doubt-answer stream feels instant.
  • Two queues at the AI worker — high priority (live class) and low priority (recordings, async questions). Live class always wins.
  • Voice doubt path — student records, browser streams chunks to AI worker, worker runs IndicWhisper (AI4Bharat's Whisper variant) to transcribe, then routes through the same RAG pipeline as text doubts.
  • Streaming TTS for voice answers — answer streams to the browser as audio chunks; student hears the AI teacher reply, not text.
  • Cohort answer caching — if Student A asks "Why is X?" and Student B asks the same thing 10 minutes later, the embedded query matches above a threshold and Student B sees Student A's answer immediately (with a "Vihaan already answered this" note). Saves cost AND feels magical.
  • Doubt dedup window — within a class, repeated identical questions are collapsed into a single doubt with an upvote count. The teacher sees the most-upvoted unanswered doubt.

09 Quiz & notes generation

Quiz generation
Phase 1

Runs after script generation. Single LLM call with the lesson script + difficulty distribution (4 easy, 3 medium, 3 hard typical) + the locked output schema (JSON with question, options, correctAnswer, explanation). Output is validated; if any question fails validation (e.g., correctAnswer not in options), the worker retries once, then escalates to a human review queue.

Every question is also self-checked: the LLM is asked "answer this question yourself" in a separate call, and if its answer doesn't match the correctAnswer, the question is flagged.

Notes generation
Phase 1

LLM call with the lesson script + a structured template (intro, key concepts, worked example, types/cases, balanced-equation block, summary chips). Output is HTML (no markdown — keeps formatting deterministic). Notes are saved to LessonNote table; the Playwright PDF renderer can produce a downloadable PDF from the HTML on demand.

10 Memory Deck cards

The AI worker also generates spaced-repetition flashcards from each lesson — definitions, formulas, key facts, common mistakes.

  • Single LLM call: "from this lesson script, extract 8-12 flashcards. Front = the prompt; back = the answer. Tag with chapter and concept type."
  • Output validated against schema, then deduplicated against the student's existing Memory Deck (via embedding similarity ≥0.85 means "you already have a card for this").
  • New cards inserted with state NEW, due immediately.
  • The spaced-repetition scheduling (SM-2 algorithm) lives in the NestJS API, not here — it just runs SQL math on the existing rows.

11 Catch-up plan generator

For a student who joined late or fell behind, the AI worker produces a daily catch-up plan.

  • Inputs: student's current lesson index per subject vs cohort's index; available hours per day (default 1.5h on weekdays, 3h on weekends); the student's quiz performance so far (to identify weaker subjects).
  • Single LLM call returns a JSON plan: [{date, lessons: [{subject, chapter, estimated_minutes}]}].
  • Heuristic constraint check after the LLM: no more than 3 lessons per day; weaker subjects get more sessions; always front-load the subject the cohort is currently on so the student is closest to live by the time they're caught up.
  • The dashboard catch-up panel reads from this plan.

12 Phase 2 — fine-tune trigger

Around 500-1,000 paying users, we'll have enough real student doubts (~20,000+) and graded answers to fine-tune a small open model on Ananta's voice.

When to start Phase 2

Triggers (ALL must be true):

  • ≥500 paying users for ≥30 days.
  • ≥20,000 doubt-answer pairs with quality signal (thumbs).
  • Daily LLM API spend exceeds ₹2,000 (a level that justifies the engineering effort).
  • Identified at least 3 systematic failure modes the off-the-shelf model has (e.g., over-formal Kanglish, hallucinated chapter references, length drift).
What Phase 2 looks like
  • Base model: Mistral 7B Instruct or Llama 3.1 8B Instruct — small enough to fine-tune on a single A100, large enough to retain reasoning.
  • Method: LoRA fine-tune (adapters, not full weights). Training cost: ~₹40k-80k per run on RunPod or Vast.ai.
  • Dataset: Filtered to high-quality answers (positive thumbs OR no negative thumb within 48h). Augmented with persona-instruction examples to lock teacher voice.
  • Deployment: Self-host on a Hetzner GPU box (~₹15-25k/month for a 24GB GPU). Serve via vLLM for batching.
  • Keep Groq as fallback for traffic overflow and for the harder reasoning questions where the small model isn't enough.
Why not skip straight to a 70B fine-tune?
Cost: 70B fine-tuning is ~10x more expensive per run, and inference needs 4x the GPU memory. At Phase 2 scale (~₹3 lakh MRR) that's not yet a comfortable spend. The 7B-8B tier is the sweet spot — most of the quality, a fraction of the cost.

13 Phase 3 — custom 70B (5k+ users)

Past 5,000 paying users (~₹30 lakh MRR), the financials support a serious custom model.

  • LoRA-fine-tune Llama 3.3 70B (or its successor) on the full accumulated doubt dataset (~200k pairs by then) + lesson scripts + Memory Deck cards.
  • Own inference cluster: 2-3 H100 GPUs on Hetzner or Oracle. ~₹3-5 lakh/month.
  • Continuous learning: monthly retrain from new data. CI pipeline auto-runs evals after each retrain; only promote to production if quality metrics improve.
  • This is also the stage to consider training a small foundation model from scratch on Indic + Karnataka-syllabus data — if a research-grade founding engineer or external researcher partnership materialises. Otherwise stick with LoRA on Llama and stay practical.

15 Quality, safety, evals

The eval suite (lightweight in Phase 1, expanded later)
  • Golden Q&A set — 200 hand-curated KSEEB questions with verified answers. Run after every prompt change. CI fails if accuracy drops >3%.
  • Persona-fidelity check — a separate LLM judges "is this answer in Vihaan's voice?" against a rubric. Sampled daily.
  • Hallucination rate — sample 100 answered doubts daily, manually check for hallucinated chapter references or false facts. Target <2%.
  • Citation accuracy — every answer cites its source chunk; we sample 50/day and check the citation is correct.
  • Refusal calibration — model should refuse off-topic questions (homework helping students cheat, etc.) but answer borderline study-related questions. Quarterly review.
The honest version: no AI system serving high-school students should run unattended. Every Sunday, priyanka (or whoever is on rotation) spends 30 minutes scrolling the past week's lowest-rated doubt answers and triaging. That's the cheapest, highest-leverage quality control we have.
