01 Where 3D lives — and where it doesn't

3D is a teaching tool first, decoration second. We earned the right to use it by picking carefully where it appears.

Where 3D lives
  • Lesson stages — the main "what is being taught right now" panel during a live class. Every subject. The exploding cube for Maths, the molecule for Science, the falling Bastille for History, the staged village for Kannada vachana.
  • Subject cards on the landing page — six small scenes that prove 3D works for every subject, not just Science. Single moment of "oh, this is different".
  • Post-class quiz reward animation — a tasteful 3D confetti / particle moment after the last question.
  • Memory Deck "review-complete" moment — when the student clears the day's queue, a small 3D "well done" scene.
Where 3D does NOT live
  • Hero of the landing page — clean editorial typography. Tried it with 3D; it made the headline fight for attention. See Ananta_Landing_3D.html v5 changelog.
  • The teacher pavilion — replaced with circular SVG anime portraits + bio cards. Reads as "real people", loads instantly.
  • Dashboard, profile, parent dashboard, admin — these are utility surfaces. 3D would slow them down and add nothing.
  • Forms, auth, settings — never.

Rule of thumb: if you can't explain in one sentence what the 3D teaches the student, it doesn't go on the page.

02 The stack (production)

The HTML mockups load Three.js r128 from a CDN. The real Next.js app uses react-three-fiber + drei + GSAP + Lenis, all from the workspace.

03 Performance budget

The numbers we measure and refuse to cross. These are for the slowest device we target: a 2-year-old Redmi at 25 Mbps 4G in Davanagere.

Enforcement: a GitHub Action runs next-bundle-analyzer on every PR and fails if any route crosses its budget. The number is in .budget.json; bumps require a comment justifying it.

Other hard limits

  • Draw calls per scene: ≤ 60
  • Triangles on screen: ≤ 100k
  • Texture size: ≤ 2048×2048; prefer BasisU compressed
  • Lights: ≤ 3 (one sun + two soft fills)
  • Shadow casters: ≤ 4 dynamic; bake everything else
  • Frame budget: 16.6 ms/frame at 60fps target — degrade to 30fps before dropping any visuals

04 Lite mode — the fallback

Some devices shouldn't get 3D at all. We detect them on first paint and serve a 2D rendition that teaches exactly the same content.

When Lite mode kicks in (automatic)
  • prefers-reduced-motion: reduce — respect OS-level setting
  • WebGL2 unavailable
  • Devicememory < 2 GB (Navigator API)
  • navigator.connection.effectiveType === "slow-2g" or "2g"
  • User toggled it manually (settings → "Lite mode")
What Lite mode looks like
  • Lesson stage: a sequence of static SVG diagrams + text captions. Same teaching beats; no Canvas.
  • Subject cards: flat illustrations; no animation.
  • Post-class reward: CSS confetti, no 3D.
  • Memory Deck celebrate: simple checkmark animation.

Each scene file exports two components: { ThreeD } and { Lite }. A single hook chooses which to render:

// src/three/use-3d-mode.ts
export function use3dMode(): "three" | "lite" {
  const [mode, setMode] = useState<"three" | "lite">("three");
  useEffect(() => {
    const reduce = matchMedia("(prefers-reduced-motion: reduce)").matches;
    const noWebGL = !window.WebGL2RenderingContext;
    const lowMem  = (navigator as any).deviceMemory < 2;
    const slow    = ["slow-2g", "2g"].includes(
      (navigator as any).connection?.effectiveType
    );
    if (reduce || noWebGL || lowMem || slow) setMode("lite");
  }, []);
  return mode;
}

05 Lesson 3D content pipeline

Two production paths: procedural (cheap) and authored glTF (when a scene is too complex to generate).

Procedural scenes (default)
  • Author the scene as apps/web/src/three/scenes/<name>.tsx using react-three-fiber primitives.
  • The AI worker generates a small JSON "scene script" per lesson (timing, captions, what to animate when). The React scene reads that script and orchestrates.
  • No model downloads. Bundle cost: ~20 KB JS per scene.
  • Use this for: cube unfolds, molecule assembly, supply/demand curves, electric circuits, simple geometry, abstract systems.
Authored glTF (heavy scenes)
  • Modelled in Blender, exported as glTF 2.0 + Draco, uploaded to Cloudflare R2.
  • Loaded via drei's useGLTF with a Suspense boundary so the React tree doesn't block.
  • Target ≤ 400 KB per asset after Draco compression.
  • Use this for: Bastille fortress, anatomy (heart/lungs/cell), historical monuments, anything organic.
Process for adding a new lesson scene: (1) write the scene component, (2) add a Lite-mode 2D fallback in the same file, (3) add a Storybook story so the QA can preview it, (4) run the budget check, (5) PR.

06 Asset rules

  • Always Draco-compress glTF. Use gltfjsx --transform in the pipeline.
  • Textures: BasisU/KTX2 where possible (3-10× smaller than PNG, decompresses on the GPU).
  • One material per mesh where reasonable — material swaps are cheap; multi-material meshes are draw-call killers.
  • Bake lighting for anything static (a fortress wall) into the texture; only dynamic things receive runtime shadows.
  • Assets live in R2, not in the JS bundle. The bundle has only the scene logic.

08 When to break these rules

A rule that can't be broken is dogma. Here are the legitimate exceptions:

  • Marketing launch landing — once we have paying users, a high-fidelity Awwwards-ambition landing page is justified. Different budget, different file, served separately from the app.
  • Special-occasion lesson — board exam eve, founder's note. We can break the lesson budget if the asset is opt-in and clearly labelled "experimental".
  • A teacher's signature opener — Vihaan walks into frame in a glTF model on his first lesson of the year. Worth 600 KB once.

Anything beyond these requires a written exception in the PR description explaining what we're trading off.
